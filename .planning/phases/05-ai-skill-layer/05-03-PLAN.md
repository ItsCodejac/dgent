    - name: "strip-noise-comments", phase: "pre-commit", type: "fix", defaultEnabled: false
    - Removes comments that restate the next line of code verbatim or near-verbatim
    - Heuristic: extract meaningful words from comment, check if the next non-blank line contains the same words
    - "Near-verbatim" threshold: if 70%+ of the comment's meaningful words appear in the next line
    - Strip common filler: "the", "a", "an", "is", "are", "to", "of", "for", "by", "in", "this"
    - Only applies to single-line comments (// or #), not block comments
    - Preserves comments that add information not in the code (e.g., "# connect to the REPLICA database" before `db.connect(url)` — "REPLICA" isn't in the code)

  <name>Task 2: Implement strip-obvious-docstrings and flag-log-bracketing with fixtures</name>
  <files>src/rules/strip-obvious-docstrings.ts, src/rules/flag-log-bracketing.ts, test/fixtures/strip-obvious-docstrings/*.ts, test/fixtures/flag-log-bracketing/*.ts</files>
  <action>
    **strip-obvious-docstrings:**
    - name: "strip-obvious-docstrings", phase: "pre-commit", type: "fix", defaultEnabled: false
    - Removes docstrings on functions under a complexity threshold where the signature is self-documenting
    - Heuristic: function body is <= 5 lines, function name + params describe behavior, docstring just restates them
    - Detects: JSDoc (/** */), Python docstrings (""" """), and # docstring blocks
    - Only removes if docstring content is extractable from the function signature
    - Conservative — if unsure, keep the docstring

    Fixtures:
    - obvious-jsdoc.input.ts / .expected.ts — trivial function with `/** Gets the user by ID */` over `function getUserById(id)` (stripped)
    - useful-jsdoc.input.ts / .expected.ts — docstring with @throws or @example info (preserved)
    - complex-function.input.ts / .expected.ts — long function with docstring (preserved — over threshold)

    **flag-log-bracketing:**
    - name: "flag-log-bracketing", phase: "pre-commit", type: "flag", defaultEnabled: false
    - Flags console.log/print statements that bracket operations with no purpose beyond narrating
    - Pattern: console.log("Doing X") followed by code, followed by console.log("Done X") or console.log("X complete")
    - Heuristic: two log statements within 5 lines of each other, one before an operation and one after, with similar content
    - Does NOT flag standalone log statements or error logging

    Fixtures:
    - bracketed-logs.input.ts / .expected.ts — log before and after a function call (flagged)
    - standalone-log.input.ts / .expected.ts — single log statement (not flagged)
    - error-logging.input.ts / .expected.ts — console.error in catch block (not flagged)

    Register both in rules array.
  </action>
  <verify>`dgent test --rule strip-obvious-docstrings` and `dgent test --rule flag-log-bracketing` pass all fixtures</verify>
